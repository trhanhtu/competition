1. Kỹ thuật Automation

Tăng Độ Ổn định (Readiness & Self-Healing)

Cơ chế "View Sẵn Sàng"

Mọi View đều kế thừa từ BaseView, buộc phải có hàm waitForReady(). Test sẽ không chạy cho đến khi View xác nhận đã tải xong. 


// BaseView.ts
export abstract class BaseView {
  abstract waitForReady(): Promise<void>;
}

// VoucherView.ts
export class VoucherView extends BaseView {
  async waitForReady(): Promise<void> {
    // Chờ API list voucher & button TẠO
    await this.page.waitForResponse("**/api/v1/voucher");
    await expect(this.createShopVoucherButton).toBeVisible();
  }
}

Chiến lược Locator Tự phục hồi

Sử dụng chiến lược .or() của Playwright để cung cấp các locator dự phòng khi UI thay đổi. 

// LoginView.ts
public get emailInput(): Locator {
  // Thử ID trước (nhanh nhất)
  return SelfHealingLocator(this.page, [
                '#normal_login_username',
                // Nếu không, thử bằng 'name'
                'input[name="username"]',
                // Cuối cùng, thử bằng placeholder
                '[placeholder="Email"]',
            ])
}

2. Bao phủ Scenario & 3. Tư duy QC

Đo lường Hiệu năng & Giả lập Rủi ro

Giám sát Hiệu năng API

Tận dụng waitForResponse để đo thời gian và assert performance, đảm bảo API không bị chậm. 

// VoucherView.ts
(status, json, resp) => {
  const timings = resp.request().timing();
  const responseTimeMs = timings.responseEnd - timings.requestStart;

  // Đảm bảo API trả về dưới 3 giây
  expect(responseTimeMs).toBeLessThan(3000);
}

Chủ động Giả lập Rủi ro

Chúng tôi có test case riêng, dùng page.route() để giả lập BE trả về 503 (Service Unavailable). Phục vụ tốt trong việc giả lập tình huống. 

// voucher.be-failure.spec.ts
await page.route('**/api/v1/voucher?*', route => {
  route.fulfill({
    status: 503,
    body: JSON.stringify({ message: 'Simulated BE Error' }),
  });
});

// Test sẽ fail vì lỗi BE
await expect(response.status()).toBe(200);

4. Sáng tạo & Custom Enhancement

Tính năng Sáng tạo: Tự động Gán lỗi (AI Triage)

Phân tích Lỗi Thông minh

Reporter không chỉ báo lỗi, mà còn *phân tích* lỗi để gán đúng người: 

// GoogleChatReporter.ts
determineResponsibleRole(errorMessage) {
  const error = errorMessage.toLowerCase();

  // Lỗi liên quan UI/Locator
  if (['locator', 'visible', 'timeout'].some(key => error.includes(key))) {
    return Role.FE; // Gán cho Frontend
  }

  // Lỗi liên quan API/Network
  if (['api', 'status', '503', '400'].some(key => error.includes(key))) {
    return Role.BE; // Gán cho Backend
  }

  // Lỗi không rõ, gán cho QC
  return Role.QC;
}

Bản đồ Tổ chức (Squad Map)

Kết hợp với bản đồ Squad, hệ thống ping chính xác người chịu trách nhiệm. 

// 1. Lỗi ở feature 'voucher'
const feature = 'voucher';

// 2. Tra cứu: 'voucher' -> Squad.GROWTH
const squad = FEATURE_TO_SQUAD_MAP[feature];

// 3. Tra cứu: Lỗi là 'locator' -> Role.FE
const role = Role.FE;

// 4. Kết quả: Ping @FE Leads của Squad.GROWTH
const leads = SQUAD_LEAD_MAP[squad][role];

4. Sáng tạo & Ứng dụng AI

Tối ưu Teamwork & Quản lý "Test Debt"

Quản lý Test với test.fixme()

Khi một test fail do một bug đã biết (đang được BE/FE sửa), chúng ta không xóa test. Chúng ta dùng test.fixme(). 

// test.fixme()
// Test này sẽ bị BỎ QUA khi chạy
// và được đánh dấu là 'fixme' trong report
test.fixme('Tạo voucher bị lỗi 500 - BE đang sửa', async ({ page }) => {
  // ... test logic
});

Lợi ích: Giúp CI/CD pipeline vẫn "XANH". Report sẽ nhắc nhở team về các "món nợ" (test debt) cần được sửa sau khi bug được fix.
